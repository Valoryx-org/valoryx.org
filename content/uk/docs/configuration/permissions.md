# Ролі та права доступу

DocPlatform використовує контроль доступу на основі ролей (RBAC) на базі Casbin — авторизаційного рушія, що працює в процесі. Перевірка прав доступу виконується менш ніж за 0,1 мс без зовнішнього сервісу.

## Ієрархія ролей

DocPlatform визначає 6 ролей у строгій ієрархії. Вищі ролі успадковують усі права нижчих ролей.

```
SuperAdmin          ← Full platform access (all workspaces)
    │
WorkspaceAdmin      ← Manage workspace settings, git config, theme
    │
Admin               ← Manage members, assign roles
    │
Editor              ← Create, edit, delete pages
    │
Commenter           ← View pages, leave comments
    │
Viewer              ← View pages only
```

## Призначення ролей

### Перший користувач

Перший зареєстрований користувач на новому екземплярі DocPlatform автоматично отримує роль **SuperAdmin**. Це відбувається лише один раз — наступні реєстрації не отримують ролі в робочому просторі, доки їх не запросять.

### Учасники робочого простору

При запрошенні користувача до робочого простору вкажіть його роль:

**Веб-інтерфейс:** Workspace Settings → Members → Invite → оберіть роль

**API:**

```bash
curl -X POST http://localhost:3000/api/v1/workspaces/{id}/invitations \
  -H "Authorization: Bearer {token}" \
  -H "Content-Type: application/json" \
  -d '{
    "email": "user@example.com",
    "role": "editor"
  }'
```

### Роль за замовчуванням

Встановіть роль за замовчуванням для нових учасників, що приймають запрошення без конкретно призначеної ролі:

```yaml
# .docplatform/config.yaml
permissions:
  default_role: viewer
```

Доступні значення: `viewer`, `commenter`, `editor`, `admin`, `workspace_admin`

## Контроль доступу на рівні сторінок

Перевизначте права доступу на рівні робочого простору для окремих сторінок за допомогою frontmatter.

### Рівні доступу (веб-редактор — внутрішні користувачі)

Для автентифікованих користувачів у веб-редакторі доступ на рівні сторінки обмежує видимість за роллю:

| Рівень | Поведінка |
|---|---|
| `public` | Будь-який учасник робочого простору може переглядати |
| `workspace` | Будь-який учасник робочого простору може переглядати (те саме, що `public` для автентифікованих користувачів) |
| `restricted` | Лише користувачі з ролями, переліченими в `allowed_roles`, можуть переглядати |

### Приклади

**Публічна сторінка** (за замовчуванням):

```yaml
---
title: Getting Started
access: public
---
```

**Обмежена лише для адміністраторів:**

```yaml
---
title: Infrastructure Runbook
access: restricted
allowed_roles: [admin, workspace_admin]
---
```

### Що означає «restricted»

Коли сторінка має `access: restricted`:

- Користувачі без необхідної ролі **не можуть переглядати** сторінку
- Сторінка **не з'являється** в результатах пошуку для неавторизованих користувачів
- Прямий доступ за URL повертає **403 Forbidden**

### Доступ до опублікованої документації

Для **опублікованого документаційного сайту** (`/p/{slug}/...`) контроль доступу працює інакше:

- Усі опубліковані сторінки є **публічними за замовчуванням** — вхід не потрібен
- Щоб вимагати вхід для всього опублікованого сайту, встановіть [`PUBLISH_REQUIRE_AUTH=true`](environment.md) — це застосовується до всіх сторінок у всіх робочих просторах
- Контроль доступу на рівні окремих сторінок в опублікованій документації (наприклад, одна сторінка лише для робочого простору, інші публічні) заплановано для майбутнього релізу

> У v0.5 поле frontmatter `access` зберігається та доступне для майбутнього використання, але не застосовується на опублікованих маршрутах. Використовуйте `PUBLISH_REQUIRE_AUTH` для обмеження доступу на рівні всього сайту.

## Внутрішні рівні ролей

Для довідки, кожна роль відповідає числовому рівню. Вищі рівні успадковують усі права нижчих рівнів:

| Роль | Рівень | Мінімальна дія |
|---|---|---|
| Viewer | 10 | `read` |
| Commenter | 20 | `read` |
| Editor | 30 | `read`, `write`, `delete` |
| Admin | 40 | `read`, `write`, `delete`, `admin` (управління учасниками) |
| WorkspaceAdmin | 50 | Усі дії робочого простору |
| SuperAdmin | 60 | Усі дії платформи (обходить усі перевірки) |

Дії мають мінімальні рівні: `read` вимагає рівень 10+, `write` вимагає 30+, `delete` вимагає 30+, `admin` вимагає 50+. Рівень ролі користувача порівнюється з мінімальним рівнем дії.

## Як оцінюються права доступу

```
API Request
    │
    ▼
Auth Middleware
(extract JWT, identify user)
    │
    ▼
Permission Middleware
(Casbin check: user + role + resource + action)
    │
    ├── Allowed → proceed to handler
    │
    └── Denied → 403 Forbidden
```

### Послідовність оцінки

1. **Вилучення ідентичності користувача** з JWT токена доступу
2. **Пошук ролі користувача** для цільового робочого простору
3. **Перевірка прав на рівні робочого простору** — чи дозволяє роль цю дію?
4. **Перевірка доступу на рівні сторінки** — якщо сторінка має `access: restricted`, чи входить роль користувача в `allowed_roles`?
5. **Повернення результату** — дозволено або відмовлено

### Продуктивність

| Метрика | Значення |
|---|---|
| **Рушій** | Casbin (у процесі, у пам'яті) |
| **Час оцінки** | < 0,1 мс на перевірку |
| **Кеш** | Версіонований (автоматично інвалідується при зміні ролі або прав) |
| **Зберігання політик** | SQLite (завантажується в пам'ять при старті) |

## Кешування прав доступу

Політики Casbin завантажуються з SQLite у пам'ять при запуску сервера. Зміни ролей або frontmatter декларацій доступу запускають інвалідацію кешу:

1. Адміністратор змінює роль користувача → версія кешу прав інкрементується
2. Редактор оновлює frontmatter сторінки з новими `access` або `allowed_roles` → кеш інвалідується для цієї сторінки
3. Наступна перевірка прав завантажує свіжу політику з SQLite

Кеш версіонований, а не часовий — вікна застарілих прав немає.

## Типові шаблони

### Публічна документація лише для читання з обмеженими внутрішніми сторінками

```yaml
# Most pages: default
access: public

# Internal pages: restricted
---
title: Incident Response Playbook
access: restricted
allowed_roles: [admin, workspace_admin]
---
```

### Редактор створює, адміністратор публікує

1. Встановіть `publishing.default_published: false` в конфігурації робочого простору
2. Редактори створюють та редагують сторінки (неопубліковані за замовчуванням)
3. Адміністратори переглядають та перемикають `published: true`

### Робочі простори для окремих команд

Створіть окремі робочі простори для кожної команди з незалежними списками учасників:

- Робочий простір `eng-docs` → інженерна команда
- Робочий простір `product-docs` → продуктова команда
- Робочий простір `internal-wiki` → усі

SuperAdmin має доступ до всіх робочих просторів для міжкомандної видимості.

## Обмеження Community Edition

Community Edition має наступні обмеження ресурсів:

| Ресурс | Ліміт |
|---|---|
| Користувачі з роллю Editor або вище | 5 |
| Робочі простори | 3 |
| Viewer та Commenter | Необмежено |
| Сторінки | Необмежено |

Ці обмеження жорстко закодовані (ліцензійний ключ не потрібен). Viewer та Commenter ніколи не враховуються у ліміті редакторів. Коли ліміт редакторів досягнуто, нових користувачів все ще можна запрошувати як Viewer або Commenter.

## Усунення несправностей

### «403 Forbidden» на сторінці, до якої я маю мати доступ

1. Перевірте свою роль: Profile → Workspace Membership
2. Перевірте frontmatter сторінки: чи включає `access: restricted` + `allowed_roles` вашу роль?
3. Попросіть адміністратора робочого простору перевірити призначення вашої ролі

### Зміни прав не набувають чинності

Зміни прав мають бути миттєвими (інвалідація кешу синхронна). Якщо це не так:

1. Вийдіть та увійдіть знову (оновіть JWT токени)
2. Перевірте логи сервера на предмет помилок інвалідації кешу
3. Запустіть `docplatform doctor` для перевірки стану системи прав

### Перший користувач не є SuperAdmin

Це трапляється, якщо перший користувач реєструється, коли база даних уже містить записи користувачів (наприклад, від попереднього встановлення). Для виправлення:

1. Зупиніть сервер
2. Видаліть базу даних: `rm {DATA_DIR}/data.db`
3. Запустіть сервер та зареєструйтесь знову

Це скидає всі дані. Використовуйте лише на свіжих інсталяціях.
